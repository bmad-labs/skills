# Content Mapping Reference

How to transform BMAD artifacts into GitHub issues, fields, labels, and milestones.

---

## 1. Issue Title Format

**Format:** `{story_id} {short_title}`

### Rules

- Prefix with story ID (`1.1`, `2.1`, etc.) for sorting and quick identification.
- Keep under 60 characters after the ID (board view truncates long titles).
- Drop redundant words from filenames (e.g., "and Package Skeletons" is dropped).
- No "Story" prefix -- the ID already signals it is a story.
- Use em dash (---) for subtitle separation, not colon (cleaner on board cards).

### Examples

| Story File | Issue Title |
|------------|-------------|
| `1-1-set-up-project-structure.md` | `1.1 Set Up Project Structure` |
| `2-1-implement-core-feature.md` | `2.1 Implement Core Feature` |
| `3-1-add-monitoring-and-alerts.md` | `3.1 Add Monitoring and Alerts` |
| `4.1-integrate-external-api.md` | `4.1 Integrate External API` |
| `6-1-build-web-dashboard.md` | `6.1 Build Web Dashboard` |
| `12-4-docker-containerization-ci-cd.md` | `12.4 Docker Containerization + CI/CD` |

## 2. Issue Body Template

Generate this exact markdown structure for each story issue.

````markdown
> **As a** {role}, **I want** {goal}, **so that** {benefit}.

## Acceptance Criteria

- [ ] AC1: {one-line summary}
- [ ] AC2: {one-line summary}
- [ ] AC3: {one-line summary}

## Tasks

- [ ] Task 2: {title}
  - [ ] 2.1: {subtask}
  - [ ] 2.2: {subtask}
- [ ] Task 3: {title}
  - [ ] 3.1: {subtask}
  - [ ] 3.2: {subtask}
- [ ] Task 4: {title}

## Dependencies

Depends on: #{issue_number_1}, #{issue_number_2}
Blocks: #{issue_number_3}

<details>
<summary>Dev Notes</summary>

{Selected dev notes: tables, code patterns, architecture decisions.}
{NOT the full story file -- only what a developer needs while working.}

</details>

---
<sub>Synced from <code>{story_file_path}</code> | Last sync: {ISO_timestamp}</sub>
````

### Section Notes

| Section | Content | Source |
|---------|---------|--------|
| Blockquote | User story from `## Story` section | Story file |
| Acceptance Criteria | One-line summaries as `- [ ]` checklist (see Section 3) | Story file `## Acceptance Criteria` |
| Tasks | Human-actionable tasks only — read each task and exclude any that are AI agent instructions (e.g., "Read standards", "Load skill"). Nested checkable list. | Story file `## Tasks / Subtasks` |
| Dependencies | `Depends on: #N, #M` with GitHub issue cross-references | Story file `## Dev Notes > Dependencies` |
| Dev Notes | Collapsible. Include tables, architecture constraints, communication patterns. Exclude code skeletons, ASCII trees, "What NOT to Do" lists | Story file `## Dev Notes` |
| ~~Standards Checklist~~ | **Excluded.** Task 1 is AI agent instructions (read standards, load skills) — not human work. | Story file Task 1 |
| Footer | Sync metadata in `<sub>` tag | Generated by sync tool |

## 3. AC Summarization Rules

BMAD acceptance criteria use verbose Given/When/Then format. Convert each AC to a single scannable line for the GitHub checklist. The full Given/When/Then stays in the local story file only.

### Approach

1. Extract the core assertion from **Then** (and **And** if present).
2. Include key parameters or thresholds inline.
3. Preserve technical identifiers (topic names, command names, rates).
4. Keep FR/NFR references inline in parentheses when present in the source.

### Before/After Examples

**Example 1 (simple command check):**

- **BMAD:** Given the project dependencies are installed, When I run `npm test` in the workspace, Then all test suites pass with zero failures.
- **GitHub:** `npm test` passes with zero failures

**Example 2 (long Given/When/Then):**

- **BMAD:** Given the server is running on port 3000, When I send a POST request to `/api/users` with a valid payload, Then the server responds with HTTP 201 and the created user object including an auto-generated ID.
- **GitHub:** POST `/api/users` with valid payload returns 201 + created user with auto-generated ID

**Example 3 (multi-clause with And):**

- **BMAD:** Given the system has 80% disk capacity remaining, When a batch job is submitted with 500MB of data, Then the system estimates processing time, And if estimated time exceeds the SLA threshold, the system refuses to start and displays a warning.
- **GitHub:** Batch job: estimates processing time; refuses to start if estimated time exceeds SLA threshold

**Example 4 (short AC):**

- **BMAD:** Given the service is running, Then health check endpoint responds on `/health` at >= 1 Hz.
- **GitHub:** Health check endpoint responds on `/health` at >= 1 Hz

## 4. Content Exclusion List

These sections from BMAD story files are **excluded** from the GitHub issue body. They remain only in local files.

| Excluded Section | Reason |
|------------------|--------|
| **Agent-only tasks** (any task number) | Tasks whose subtasks are predominantly "Read `.knowledge-base/...`" or "Load skill `...`" instructions. These are AI agent setup steps, not human work. **You (the agent) must identify these by reading the task content** — they can appear at any position, not just Task 1. Exclude the entire task and all its sub-items from the issue body. |
| `## Dev Agent Record` | Internal AI agent metadata (model, completion notes) |
| `## File List` | Internal BMAD cross-references |
| `## References` | BMAD-internal source document links |
| Full code skeleton examples (boilerplate templates, scaffold patterns) | Too verbose; developers read standards docs directly |
| "What NOT to Do" lists | Agent guard rails, not issue content |
| Package structure ASCII art (full directory trees) | Too verbose for issue; tables in Dev Notes are acceptable |
| Naming convention rule blocks | Standards content, not story content |
| `Status: ready-for-dev` line | Redundant with project Status field |
| Sprint assignment | Redundant with milestone |
| Dev assignment | Redundant with issue assignee |
| Epic grouping text | Redundant with label |

## 5. Source of Truth Table

Each field has exactly one owner. The sync direction determines whether the field is pushed from BMAD to GitHub or pulled from GitHub back to BMAD.

| Field | Owner | Direction | Notes |
|-------|-------|-----------|-------|
| Body (user story, AC, tasks) | BMAD story files | Push to GitHub | Local file is authoritative |
| Acceptance Criteria text | BMAD story files | Push to GitHub | Summarized on push |
| Task list text | BMAD story files | Push to GitHub | Task 2+ only |
| Status | GitHub Projects | Pull to BMAD | `Status: {mapped_value}` in story file |
| Sprint | BMAD `sprint-plan.md` | Push to GitHub | Sets milestone + date fields |
| Dev assignment | GitHub | Pull to BMAD | Set via issue assignee + Dev field |
| Epic | BMAD `epics.md` | Push to GitHub | Sets label + Epic custom field |
| Phase | BMAD `sprint-plan.md` | Push to GitHub | Sets label + Phase custom field |
| Task checkboxes (checked/unchecked) | GitHub | Pull to BMAD | Devs check off tasks in GitHub |
| Story Points | GitHub only | N/A | Not in BMAD files |
| Priority | GitHub only | N/A | Not in BMAD files |
| Dependencies | BMAD story files | Push to GitHub | Cross-referenced with `#N` issue numbers |

|------------------|----------------------|--------------|
| `ready-for-dev` | `Ready` | Ready |
| `in-progress` | `In Progress` | In Progress |
| `in-review` | `In Review` | In Review |
| `done` | `Done` | Done |
| _(not yet created / no status)_ | `Backlog` | Backlog |

### Parsing

The BMAD status line appears as plain text on line 3 of story files:

```
Status: ready-for-dev
```

It is NOT YAML frontmatter. Parse with: `^Status:\s*(.+)$`

## 8. Milestone Scheme

12 milestones, one per sprint. Create all milestones before syncing issues.

Milestones are dynamically derived from `sprint-plan.md`. The parser script extracts sprint numbers, dates, objectives, and story counts. Do not hardcode milestone values — always read them from the parsed sprint data.

**Example:**

| Milestone | Due Date (ISO) | Description Template |
|-----------|---------------|----------------------|
| `Sprint 1` | `{end_date}` | Sprint Objective: {objective} / Demo Date: {demo_date} / Working Days: {count} / Stories: {story_count} |
| `Sprint 2` | `{end_date}` | ... |
| ... | ... | ... |

### Milestone Description Format

```
Sprint Objective: {objective}
Demo Date: {friday date}
Working Days: {count}
Stories: {count}
```

## 9. Link-Back Strategy

After syncing an item to GitHub, write the GitHub URL back into the local file so agents can detect "already synced" and use edit mode instead of creating duplicates.

### Case 1: Story Files (NO frontmatter)

Story files do not have YAML frontmatter. The `Status:` line is plain text.

**Strategy:** Wrap the H1 title in a markdown link to the GitHub issue.

Before sync:
```markdown
# Story 1.1: Set Up Project Structure

Status: ready-for-dev
```

After sync:
```markdown
# [Story 1.1: Set Up Project Structure](https://github.com/OWNER/REPO/issues/42)

Status: ready-for-dev
```

**Detection regex:** `^# \[(.+?)\]\((.+?)\)$`

- Match group 1: title text
- Match group 2: GitHub URL
- If the H1 is a plain `# title` (no link), the item has not been synced yet.

### Case 2: Planning Files (WITH frontmatter)

Files like `epics.md` and `sprint-plan.md` have `---` YAML frontmatter.

**Strategy:** Add `gh_item_url` and `last_synced` to the YAML frontmatter.

```yaml
---
status: complete
gh_item_url: https://github.com/OWNER/REPO/issues/99
last_synced: "2026-04-01T10:30:00Z"
---
```

For `epics.md` (multiple epic tracking issues):

```yaml
---
status: complete
gh_item_urls:
  epic_1: https://github.com/OWNER/REPO/issues/100
  epic_2: https://github.com/OWNER/REPO/issues/101
  epic_3: https://github.com/OWNER/REPO/issues/102
last_synced: "2026-04-01T10:30:00Z"
---
```

**Detection logic:** If `gh_item_url` (or `gh_item_urls`) exists in frontmatter, item is already synced -- use edit mode.

## 11. Story ID Derivation

Extract the story ID from the implementation artifact filename.

### Pattern

Filenames follow two formats (inconsistent naming in the repo):

| Filename Pattern | Example | Story ID |
|-----------------|---------|----------|
| `{epic}-{story}-{slug}.md` (dash separator) | `1-1-set-up-project-structure.md` | `1.1` |
| `{epic}-{story}-{slug}.md` (dash separator) | `3-1-power-manager-node-with-range-estimation.md` | `3.1` |
| `{epic}.{story}-{slug}.md` (dot separator) | `4.1-stop-and-scan-capture-image.md` | `4.1` |
| `{epic}.{story}-{slug}.md` (dot separator) | `4.2-quality-gate-evaluate-image.md` | `4.2` |

### Extraction Logic

```
Regex: ^(\d+)[-.](\d+)-
Group 1: epic number
Group 2: story number
Story ID: "{group1}.{group2}"
```

Examples:
- `1-1-set-up-project-structure...` --> match `1`, `1` --> `1.1`
- `3-1-power-manager-node...` --> match `3`, `1` --> `3.1`
- `4.1-stop-and-scan...` --> match `4`, `1` --> `4.1`
- `12-4-docker-containerization...` --> match `12`, `4` --> `12.4`
